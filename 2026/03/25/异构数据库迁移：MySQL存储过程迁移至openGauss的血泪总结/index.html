异构数据库迁移：MySQL 存储过程迁移至 openGauss 的血泪总结

作为一名老兵，最近在处理某管理系统的国产化适配，将几十个核心存储过程从 MySQL 搬到 openGauss（基于 PostgreSQL 内核）。本以为是丝滑搬迁，结果在 Docker 环境里反复“点火”失败。
为了让兄弟们少走弯路，我把这次踩到的雷、原因以及“标准姿势”整理出来。
核心思想：别用 MySQL 的思维写 openGauss。

1. 致命的单引号：双单引号的“幻觉”

💣 踩雷现场

在动态 SQL 拼接时，经常看到这种报错：ERROR: syntax error at or near "''"

🔍 原因分析

这是最容易翻车的地方。在 openGauss 的 PL/pgSQL 中：

  • 静态 SQL 区：比如 FOR r IN (SELECT ... WHERE name = '_')，这里只需要标准单引号。
  • 动态拼接区：比如 v_sql := 'SELECT ... WHERE name = ''_'''。
  • 很多时候我们习惯性地在 FOR 循环的查询语句里也用了双单引号（转义后的单引号），结果 DDL 编译器直接罢工。

✅ 解决方案

  • 静态查询：老老实实写单引号 '_'。
  • 动态拼接：强烈建议放弃手动拼接，改用内置函数保护：
    • quote_literal(v_val)：自动处理值的单引号转义。
    • quote_ident(v_col)：自动处理列名/表名的双引号保护（防止关键字冲突）。

2. 标识符大小写：消失的 typeId

💣 踩雷现场

ERROR: column "typeid" does not exist
但我明明在表结构里定义的是 typeId！

🔍 原因分析

MySQL 对大小写相对宽容，但 openGauss（Postgres 系）非常严谨：

  • 如果不加双引号，所有的标识符（表名、列名）都会被强制转为小写。
  • 如果你的物理表定义里用了驼峰命名（如 typeId），但在 SQL 里没加双引号，它会去查 typeid，然后告诉你查无此列。

✅ 解决方案

在存储过程中，涉及到实体表字段的地方，特别是驼峰命名的，一律加双引号保护：
DELETE FROM operate_record_history WHERE "typeId" = 2;

3. 函数异构：INSTR 与 SUBSTRING_INDEX 的退场

💣 踩雷现场

直接搬运 MySQL 的字符串处理函数，系统提示函数不存在。

🔍 原因分析

openGauss 的内核函数集与 MySQL 并不完全重合。

  • INSTR 在 MySQL 里很常用，openGauss 推荐用标准的 POSITION。
  • SUBSTRING_INDEX 在 MySQL 里用来切分字符串，openGauss 里对应的核武器是 split_part。

✅ 解决方案

做一次全量函数映射替换：

  • INSTR(str, substr) → POSITION(substr IN str)
  • SUBSTRING_INDEX(str, '_', 1) → split_part(str, '_', 1)
  • GROUP_CONCAT → string_agg(field, ';')（注意：string_agg 性能更稳，且没有 1024 字节的默认限制）。

4. 游标进化：从手动控制到隐式循环

💣 踩雷现场

MySQL 里的游标需要 DECLARE -> OPEN -> FETCH -> CLOSE，还得配一个 CONTINUE HANDLER 来判定结束。代码又长又臭，还容易死循环。

🔍 原因分析

openGauss 提供了更现代的 FOR ... IN 结构。

✅ 解决方案

研发建议： 除非有极特殊的逐行处理逻辑，否则一律改写为 FOR record_var IN (SELECT ...) LOOP。
这种方式下，数据库内核会自动管理游标的生命周期和内存预取，不仅代码精简 50%，执行效率也更高。

5. DDL 的事务特性：TRUNCATE 不再是“泼出去的水”

💣 踩雷现场

在重置脚本里执行 TRUNCATE，以为像 MySQL 一样会自动提交，结果后面报错回滚后，表数据竟然“复活”了。

🔍 原因分析

在 openGauss 中，TRUNCATE 是事务安全的。MySQL 执行 TRUNCATE 会隐式提交。openGauss 里它就在当前事务里，如果整个存储过程失败，数据会回滚回来。

✅ 解决方案

这是一个好特性，但需要改变习惯。在进行全量数据重置操作时，配合 RAISE NOTICE 输出实时进度，确保你清楚当前处于事务的哪个阶段。

总结

迁移后的架构拓扑经过这次全量重构，系统的同步性能得到了显著提升，特别是利用 string_agg 和 format() 动态 SQL 后，原本容易崩溃的元数据转置逻辑现在稳定得一塌糊涂。
作为老程序员，最后叮嘱一句： 迁移不是简单的“改语法”，而是“换引擎”。拥抱 openGauss 的强类型和事务特性，你的后端逻辑会稳如老狗。
